# Markdown spec

This document describes the currently implemented features of markdown, any extensions added by violet, and any limitations of the parser. This can also serve as an intro to markdown, but only Violet's specific flavour.

## Short version

The spec is a combination of the proper markdown RFC, for obvious reasons, GitHub-flavoured extensions, and out of necessity, extensions for inja, the templating library.

It's deeply unclear to me what to best describe as the actual source of the markdown features, since many implementations have borrowed from each other, while other implementations have done the same thing in wildly different ways. Aside the inja extensions, the markdown is meant to feel like common sense coming from most of the major platforms that feature markdown.


## Core concepts

This section describes the idea behind the parser.

### Strictness

The parser tries to be lenient where possible, but due to implementation reasons, is aggressive in how certain bits of syntax are enforced.

The leniency is designed to be intentional. If something is lax, it must be intentionally implemented. This means that there are many things that may normally be technically accepted in other markdown flavours that is simply invalid.

For example, attempting to create an `h7` is a syntax error rather than a silent conversion to a paragraph. If you try to use special syntax and those are not valid, and not within the list of exceptions, you get an error message.

In the longer term, this will include an explicit message showing the context for the error along with the error message. One notable example of this is in error messages. The following is an illegal code block:

````

### Raw HTML

HTML in the markdown doc is not escaped. It's assumed the pages you render are trusted, and that they therefore may contain friendly HTML. To avoid breaking native HTML and introducing extra complexity in the template parsing pipeline, raw HTML is supported.

This also means that, in the event of parser limitations, the parser can be bypassed with raw HTML.

### UTF-8 or ascii input is required

C++ has very limited facilities in place for dealing with varied charsets. Much of the parser relies on ascii codepoints being stable in UTF-8. If your files are not ascii or UTF-8, they will likely fail to parse in all kinds of exotic ways. 

## Headers

Headers are implemented in the same way headers are elsewhere:
```markdown
# H1
## H2
### H3
#### H4
##### H5
###### H6
```

Trailing `#`s are not supported, and are rendered verbatim as text.

IDs are[^1] generated by lower-casing characters, only keeping non-latin letters and numbers (and - and _), and replacing spaces with `-`. In the event of duplicate IDs, disambiguation is done with `-{number}`. For example, given
```markdown
## C
## C++
## C#
```

The three IDs generated would be (in order)
* `c`
* `c-1`
* `c-2`

Similarly, in the event of the entire ID decaying to nothing, numbers become the sole tiebreakers
```markdown
# æøå
```
gets the id `1`. Note that this acts weirdly when spaces are involved. For example, given
```markdown
# 여우는 예뻐요
```
The ID decays to `-`, which is technically valid, though not useful.

A better solution would be nice here, but is likely too complex to implement.

Explicitly specifying IDs is not supported in the syntax. you can get around this with raw HTML.

## Bold and italics

Bold and italics are implemented using the classic stars:

* `*italic*` *italic*
* `**bold**` **bold**
* `***bold italic***` ***bold italic***

However, due to limitations in the parser, bold italics cannot be mixed. This is because the following two bits of markdown render in two completely different ways, and determining the structure turned out to be really fucking hard:

* `***bold italic* just bold**`
* `***bold italic** just italic*`

In v1, this is not going to be supported. However, you can bypass this with some extra spaces. This is perfectly valid:

> ```markdown
> ** *bold italic* just bold**
> * **bold italic** just italic*
> ```

Supporting mixed emphasis with three opens is a goal eventually, but not right now. The space variants will always be supported.

## Inline code and code blocks

Inline code is indicated with backticks:
```markdown
`this is inline code`
```

And code blocks are implemented as the fairly popular code fences:
````markdown
```optional-language
content
```
````

**Space-indented code blocks are not supported**. Only code fences are allowed.

Like in other parsers, increasing the number of backticks is required to escape backticks within the content. For example, in the previous code block that contains the markup for a codeblock, four backticks were used in the outer block:
`````markdown
````markdown
```optional-language
content
```
````
`````

Similarly, that block used 5 backticks in the outer block.

This same logic also applies to inline code, but inline code is more limited. Similarly to bold italics, it has no way to disambiguate that 
````markdown
```code` more content``
````

is in fact just a two-backtick inline code. Additionally, three-backtick inline code at the start of a line is currently misidentified as a code block. You can get around this by prefixing the line with a space, though I would advise you to stick to one backtick since three is redundant the vast majority of the time. This will probably be fixed at some point.

### Language rendering

Languages are rendered in HTML as a highlightjs-compatible class:
```html
<pre><code class="language-markdown">... content</code></pre>
```

No further rendering is done, as implementing arbitrary language parsing is a whole different beast, and not worth the effort when plenty of equivalent web-based implementations already exists. It's required that the implementing site sources highlightjs or equivalent manually, and loads the relevant languages for the code blocks to highlight. Violet's role is just in propagating the language information.

This also means that the supported languages and their names depends on what your syntax highlighter of choice supports and uses.

## Tables

Tables are not implemented at this time. They'll require a fair few changes internally to be implementable while also supporting markdown content, and will likely end up being a mostly novel implementation, or just GFM-like. GFM tables have multiple annoyances that make me somewhat hesitant to directly use the GFM markup, particularly in dealing with multiline cells and merged cells, both vertically and horizontally. Part of me suspects tables are just  better implemented in HTML, since HTML can actually describe the table properly. That is also the only solution for rendering tables for now.


## Lists

Lists are identical to the vast majority of flavours, but are much more strict on indent.

Each level of the list is indented by precisely two spaces. Similarly, content continuations are indented with exactly two spaces:

```markdown
* Unordered list
  with a single paragraph spread across two lines
* Bullet

  with two paragraphs
* Bullet
  * With sub-bullet
    1. with sub-numbered list
```

## Blockquotes

Blockquotes are implemented with `>`
````markdown
> this is a blockquote
> ```
> this is a code block in a blockquote
> ```
>
> > This is a quote in a quote
````

A single line without a > is enough to terminate a quote:

```markdown
> This is a quote
> > This is a nested quote

> This is a separate quote
>
> This is a continuation line
```

> This is a quote
> > This is a nested quote

> This is a separate quote
>
> This is a continuation line

### Callouts

Callouts are also supported. All the GitHub types are supported, as well as the extra `info` callout type. Note that these are strictly typed, and using an invalid callout type is a compile error.
```markdown
> [!note]
> Content

> [!info]
> Content

> [!tip]
> Content

> [!important]
> Content

> [!warning]
> Content

> [!caution]
> Content
```

This is how they appear with the default theme:

> [!note]
> Content

> [!info]
> Content

> [!tip]
> Content

> [!important]
> Content

> [!warning]
> Content

> [!caution]
> Content

Note that unlike other implementations, all the parser does is provide a callout type and a callout title paragraph object. The styling has to be manually implemented.


## Links

All three standard link formats are implemented:

```markdown
# Named links
[link text](https://example.com/inline)

# Reference links
[link text][link-ref]

[link-ref]

## Reference link definition
[link-ref]: https://example.com/ref
```

Rendered (excl. headers):

> [link text](https://example.com/inline)
>
> [link text][link-ref]
>
> [link-ref]

[link-ref]: https://example.com/ref
